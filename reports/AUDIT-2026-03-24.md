## 2. Architecture Summary

### Post Publishing Flow

```
Frontend (React)
  → POST /posts (Backend NestJS)
  → Create Post rows in DB (one row per channel)
  → Temporal Workflow scheduled (one workflow per post row)
  → Task Queue routed by providerIdentifier (e.g. "x", "instagram", "facebook")
  → Integration Provider called (x.provider.ts, instagram.provider.ts, etc.)
  → Platform API called
  → Post state updated → PUBLISHED or ERROR
```

### Key Design Constraint
**Each post DB row is tied to exactly ONE integration.** When a user selects 3 channels, 3 separate Post rows are created, each with its own workflow. If one fails, the others are unaffected.

### Integration Authentication
- **X/Twitter:** OAuth 1.0a, token stored as `accessToken:accessSecret`
- **Instagram:** OAuth 2.0 via Facebook Graph API v20.0 (requires Business Account linked to a Facebook Page)
- **Facebook:** OAuth 2.0 via Facebook Graph API v20.0

## 3. Bug Report — ayaneikedatv (Facebook-only posts)

### Symptom
Posts for "ayaneikedatv" are published on Facebook, but NOT on Instagram or X.com.

### Investigation Results

There are two independent root causes, each blocking one platform:

---

### Root Cause A — X.com: Missing API Credentials

**File:** `libraries/nestjs-libraries/src/integrations/social/x.provider.ts` (line 67–69)

```typescript
return {
  appKey: process.env.X_API_KEY!,    // ← undefined if env var not set
  appSecret: process.env.X_API_SECRET!, // ← undefined if env var not set
};
```

**Confirmed:** `.env` file does not exist in the project root. The `.env.example` shows `X_API_KEY=""` and `X_API_SECRET=""` — both empty.

**What happens:** The `TwitterApi` client is instantiated with `appKey: undefined` and `appSecret: undefined`. Twitter API calls will fail with an authentication error. The error IS caught and the post SHOULD be set to ERROR state — but since the `X_API_KEY`/`X_API_SECRET` are never configured, **every X post from "ayaneikedatv" fails before reaching the API**.

**Fix required (configuration):**
1. Set `X_API_KEY` and `X_API_SECRET` in the environment/docker-compose
2. OR configure per-channel X API credentials via the channel settings UI (additionalSettings)

---

### Root Cause B — Instagram: Silent skip when integration needs reconnect

**File:** `apps/orchestrator/src/workflows/post-workflows/post.workflow.v1.0.1.ts` (line 99–122)

```typescript
if (post.integration?.refreshNeeded) {
  await inAppNotification(...);
  return;  // ← Post stays in QUEUE forever, no ERROR state set
}

if (post.integration?.disabled) {
  await inAppNotification(...);
  return;  // ← Same silent skip
}
```

**What happens:** If the Instagram integration has `refreshNeeded = true` (token expired or revoked), the workflow:
1. Sends an in-app notification ✓
2. Returns early WITHOUT calling `changeState(postId, 'ERROR', ...)` ✗
3. The post remains in **QUEUE state indefinitely** — it looks like it's pending but will never be posted

**This is a code bug.** The post should be marked ERROR so the user can see it failed.

**Fix applied:** See section 4 below.

---

### How to Verify (SQL)

```sql
-- Check all integrations for ayaneikedatv
SELECT id, "providerIdentifier", disabled, "refreshNeeded", name
FROM "Integration"
WHERE name = 'ayaneikedatv';

-- Check post states for recent posts
SELECT p.id, p.state, p.error, i."providerIdentifier", p."publishDate"
FROM "Post" p
JOIN "Integration" i ON p."integrationId" = i.id
WHERE i.name = 'ayaneikedatv'
ORDER BY p."publishDate" DESC
LIMIT 20;
```

**Expected findings:**
- Facebook posts: `state = PUBLISHED`
- Instagram posts: `state = QUEUE` (stuck — due to Bug B above)
- X posts: `state = ERROR` (failed due to missing credentials)

## 4. Critical Bugs Found

### BUG-001 — Workflow silently skips posts when integration needs reconnect

**Severity:** CRITICAL
**File:** `apps/orchestrator/src/workflows/post-workflows/post.workflow.v1.0.1.ts`
**Lines:** 99–122

**Problem:** When `refreshNeeded=true` or `disabled=true`, the post workflow returns without calling `changeState(postId, 'ERROR', ...)`. The post remains in QUEUE state permanently and is never retried or surfaced as failed.

**Fix:** Call `changeState()` before each early return.

**Status:** ✅ FIXED in this session (see diff below)

```typescript
// BEFORE (broken):
if (post.integration?.refreshNeeded) {
  await inAppNotification(...);
  return;  // ← silent skip
}

// AFTER (fixed):
if (post.integration?.refreshNeeded) {
  await inAppNotification(...);
  await changeState(postsList[0]?.id ?? postId, 'ERROR', new Error('Integration needs reconnect'), postsListBefore);
  return;
}
```

---

### BUG-002 — X posts fail silently when X_API_KEY / X_API_SECRET not configured

**Severity:** CRITICAL (configuration)
**File:** `libraries/nestjs-libraries/src/integrations/social/x.provider.ts` (line 67–69)
**Also:** `.env` / docker-compose / deployment config

**Problem:** `X_API_KEY` and `X_API_SECRET` are not set. `getAppCredentials()` returns `{ appKey: undefined, appSecret: undefined }`. X posting fails for ALL channels unless per-channel credentials are set via additionalSettings.

**Fix required:** Set environment variables OR configure per-channel X credentials in the Postiz UI under channel settings.

**Status:** ⚠️ PENDING — requires environment configuration

## 5. High Severity Issues

### ISSUE-001 — Token refresh is a no-op for X, Facebook, Instagram

**File:** `x.provider.ts`, `facebook.provider.ts`, `instagram.provider.ts`

All three providers implement `refreshToken()` as:
```typescript
async refreshToken(): Promise<AuthTokenDetails> {
  return { id: '', name: '', accessToken: '', refreshToken: '', expiresIn: 0, picture: '', username: '' };
}
```

**Impact:** If a token expires, the 5-retry refresh loop in the workflow succeeds (no error thrown), but the integration's token is never actually renewed. The user must manually reconnect. The `refreshNeeded` flag IS set correctly, but users may not notice the in-app notification.

**Recommendation:** Add proper token refresh logic, or at minimum throw a clear error in `refreshToken()` to avoid the misleading silent success.

---

### ISSUE-002 — Instagram auth may authenticate against Facebook user instead of Instagram Business Account

**File:** `libraries/nestjs-libraries/src/integrations/social/instagram.provider.ts` (lines 357–412)

The `authenticate()` method returns `id` and `name` from `GET /me` (Facebook user profile), not from the Instagram Business Account. The integration `internalId` is set to the Facebook user ID.

When `post()` is called, it uses the Facebook Page token to post to the Instagram Business Account. If the Facebook Page does not have an Instagram Business Account linked, posting will fail.

**Recommendation:** During `authenticate()`, validate that at least one Facebook Page with an `instagram_business_account` exists, and surface a clear error if not.

---

### ISSUE-003 — Instagram media upload has no timeout

**File:** `libraries/nestjs-libraries/src/integrations/social/instagram.provider.ts` (lines ~584–596)

The media upload polling loop checks status every 30 seconds with no maximum retry count or timeout. If Facebook's media processing gets stuck, the loop runs indefinitely.

**Recommendation:** Add a max retry count (e.g., 20 attempts = 10 minutes) and throw `BadBody` if exceeded.

## 6. Medium Severity Issues

### ISSUE-004 — Uncontrolled console.log with potentially sensitive data

**Files:**
- `apps/backend/src/api/routes/posts.controller.ts` line ~163: logs full POST body
- `apps/backend/src/api/routes/public.integrations.controller.ts`: logs full request body
- `apps/orchestrator/src/workflows/send.email.workflow.ts`: logs error objects
- `apps/backend/src/api/routes/media.controller.ts`: `console.log('hello')`
- `libraries/nestjs-libraries/src/integrations/social/instagram.provider.ts` lines 529, 581, 597: logs media IDs

**Risk:** Request bodies can contain OAuth tokens, post content with PII, media URLs. Structured logging (Sentry, Winston) should be used instead.

---

### ISSUE-005 — Per-channel X credentials incomplete for OAuth flow

**File:** `libraries/nestjs-libraries/src/integrations/social/x.provider.ts`

`generateAuthUrl()` and `authenticate()` always use global env vars (`X_API_KEY`/`X_API_SECRET`). Per-channel credentials can only be added AFTER the channel is created. This creates a UX problem: the OAuth flow uses the global app, but subsequent posts use the per-channel app.

**Impact:** OAuth callback URL may be registered on a different app than the one used for posting.

---

### ISSUE-006 — Webhook failures silently swallowed

**File:** `apps/orchestrator/src/activities/post.activity.ts`

```typescript
webhooks.map(async (webhook) => {
  try {
    await fetch(webhook.url, { ... });
  } catch (e) {
    /**empty**/
  }
})
```

Failed webhook deliveries are never retried, logged, or surfaced to the user.

---

### ISSUE-007 — `additionalSettings` type validation missing

Per-channel credentials are stored as a JSON string and parsed at runtime with no TypeScript type validation. A typo in field names (e.g., "X Api Key" vs "X API Key") will silently fall back to global env vars with no warning.

## 7. Security Observations

### Positive
- JWT-based auth with secret key
- Per-organization data isolation via `organizationId`
- OAuth 2.0 / OAuth 1.0a for third-party auth
- Rate limiting via `throttler` module
- Temporal task queue isolation by provider

### Concerns
- **OAuth tokens stored in plaintext** in PostgreSQL (`Integration.token`)
- **additionalSettings (API keys) stored in plaintext** in PostgreSQL
- No credential encryption at rest — if DB is compromised, all tokens are exposed
- Console.log may leak tokens in application logs
- No HMAC signature validation on webhook deliveries
